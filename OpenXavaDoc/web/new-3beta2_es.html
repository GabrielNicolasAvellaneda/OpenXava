Estas son alguna de la nuevas característica en OpenXava 3 beta2:
  • @RowStyle se puede aplicar a colecciones
  • Módulos por defecto
  • Anotación @XOrderBy

@RowStyle se puede aplicar a colecciones

Es decir, ahora podemos declarar una colección como sigue:
@OneToMany(mappedBy="comercial")
@ListProperties("codigo, nombre, observaciones, relacionConComercial, comercial.nivel.descripcion, tipo") 
@RowStyle(style="highlight", property="tipo", value="fijo") 
private Collection<Cliente> clientes;
o como sigue:
@RowStyles({
  @RowStyle(forViews="Especiales", style="highlight", property="tipo", value="especial"),
  @RowStyle(forViews="Completa", style="highlight", property="tipo", value="fijo")     
})
private Collection<Cliente> clientes;
@RowStyle se comporta como en el caso del Tab.

Módulos por defecto

OpenXava asume un módulo por defecto para cada componente en la aplicación, aunque el módulo no se defina explícitamente en aplicacion.xml.

Es decir, si definimos un componente Factura.xml, podemos abrir nuestro navegador e ir a:

http://localhost:8080/Gestion/xava/modulo.jsp?application=Gestion&module=Factura

También un portlet es generado para permitir desplegar el módulo como un portlet JSR-168 en un portal Java.

Y todo esto sin necesidad de definirlo en aplicacion.xml.

El controlador para estos módulos por defecto será Typical, pero podemos cambiar este valor por defecto usando el elemento modulo-defecto en aplicacion.xml, de esta manera:

<aplicacion nombre="Gestion">
  <modulo-defecto>
    <controlador nombre="MantenimientoGestion"/>
  </modulo-defecto>
</aplicacion>

En este caso todos los módulos por defecto de la aplicación Gestion tendrán el controlador MantenimientoGestion asignado a ellos.

Si queremos que cierto módulo no use estos controladores por defecto, tenemos dos opciones:

  1. Definir un controlador en nuestro controladores.xml con el mismo nombre que el componente.

  2. Definir explícitamente el módulo en aplicacion.xml, tal y como se explica en la sección 8.1.

Resumiendo, si definimos un componente, llamado Cliente por ejemplo, entonces tenemos un módulo llamado Cliente, y también un portlet. Este módulo se definirá de una de la siguiente formas:

  1. Si definimos un módulo llamado Cliente en aplicacion.xml entonce este módulo será el válido, si no...

  2. Si definimos un controlador llamado Cliente en controlladores.xml un módulo será generado usando el controlador Cliente como controlador y el componente Cliente como modelo, si no...

  3. Si definimos un elemento modulo-defecto en nuestro aplicacion.xml entonces un modulo se generará usando los controladores en modulo-defecto y el componente Cliente como modelo, si no ...

  4. un módulo con Typical como controlador y Cliente como modelo se asumirá en última instancia.

Anotación @XOrderBy

@XOrderBy es una versión eXtendida de @OrderBy para usar con colecciones.

El @OrderBy de JPA no permite usar propiedades calificadas (propiedades de referencias).
OpenXava tiens su propia versión de @OrderBy, esta @XOrderBy sí que lo permite.
Ejemplo:
@OneToMany (mappedBy="factura", cascade=CascadeType.REMOVE)
@ListProperties("producto.descripcion, cantidad, precioUnitario, cantidad")
@XOrderBy("producto.descripcion desc") 
private Collection<LineaFactura> lineas;
Podemos notar como producto.descripcion puede usarse para ordenar la colección.
Para usar una propiedad calificada, esta tiene que estar incluida en el @ListProperties, como en el caso de producto.descripcion.

El orden en @XOrderBy tiene solo efeto a nivel visual, cuando accedemos por código a la colección, la colección se ordenará según lo indicado en el @OrderBy de JPA o por su order por defecto.
