Why to use OpenXava if we have Spring, Hibernate, JSF, EJB3 and so on?

Nowadays we live great times. It's true that Spring, Hibernate, JSF and EJB3 are far better than J2EE 1.4. All these frameworks and technologies are good (in fact OpenXava uses some of them), but if you need to develop business application the amount of code to write continues being big, and the productivity is low comparing with other technologies (Visual Basic, 4GL, Delphi, RPG, etc).
OpenXava simply has a higher level of abstraction in order to obtain a simple and easy way to develop business application with Java EE/J2EE in a Object Oriented fashion.
If you do not believe me. Try it. See the demo applications, for example the JBoss Seam Booking Demo developed with OpenXava, see the code for it, and try to develop it with your favourite framework, and compare  yourself.

Why OpenXava is better than a MDA solution?

We can talk about it's different and not only better. With MDA we can generate an application from UML models and in that way we reduce the costs involved in our Development. In fact with OpenXava we make the same thing but we don't start from UML models we start from XML.

Now we have advantages from this point of view beside the MDA model.

  • XML is text , the advantages are:
    • We don't need sofisticated tools, a single editor works fine.
    • There are many people than prefers text to graphical. Yes, it's true, one image is better than thousand words but is also true that the text has been a very efficient way to send information during centuries (at last the Latin was imposed to the ancient egyptian art of writing).
  • XML is extensible by nature, it's very easy to add new features to OpenXava
  • XML is very popular and has a lot of validators and special editors.
  • XML is a very known standard.
  • XML is very easy.
OpenXava and MDA aren't exclusive things, in fact you can create a MDA tool over OpenXava, only need to generate the XML files in OpenXava starting from the UML model.

Why OpenXava does not use struts?

Struts allow to define the control layer of a web application, OpenXava has his own control framework. Struts is navigation between pages oriented, while in the typical business applications the user can made a lot of actions in the same page. Struts for this last thing is something complex, and the OpenXava control layer has been constructed to do so.

Another advantage from OpenXava is that control layer isn't dependable from web technologies, we could have by example one OpenXava application with a different presentation technology (in fact there is an OpenXava version to develop Swing applications).

Many Struts Advanced Programmers are very exciting working with OpenXava to develop business applications.

It's only XML, no Java class?

OpenXava uses JavaBeans to insert your custom functionality in the application using the strategy pattern.
Moreover the upcoming OpenXava versión (3.0) will allow you to use Java classes + annotations as alternative to XML for business component definition.

How scalable it is?

OpenXava generates a standard Java EE/J2EE application, hence it's as scalable as Java EE/J2EE.
OpenXava uses EJB3 JPA, Hibernate or EJB CMP2 for persistence, and some session scope state at servlets level. The code generated by OpenXava is good, but when you write your own code you must to be wise in order to obtain a good performance, specially in batch processes, but this is an issue of J2EE not OpenXava.

What type of users manager system and navigation could you use with OpenXava?

The modules generated by OpenXava are JSR-168 portlets, hence are deployabled in any compliant Java Portal with its own security and navigation features. 
OpenXava site is powered by Liferay Portal, therefore in this case the Liferay security is used, but you can choose another portal or create your own security and navigation framework for OpenXava modules.

What Application Servers are supported?

OpenXava has been tested with:
  • Tomcat 5.5
  • JBoss 4.0.x and 3.2.x
  • WebSphere 6.1, 6.0, 5.1 
Any Servlet 2.3 compliant container can execute OpenXava applications.
If you want use EJB (optional in v2 or greater) you need at least a J2EE 1.3 application server.
If you want to use EJB3 JPA you need an application server with Java 5 support (a Java EE 5 server is not required).
WebSphere 5.0 works fine with OpenXava 1.2.1, but it's not tested since OpenXava 2.

What Java Portals are supported?

Any JSR-168 compatible portal. Although the next ones are tested:
  • Apache JetSpeed 2
  • WebSphere Portal 5.1 and 6.0
  • Liferay 4.1.x
  • Stringbeans

What Development Tools can be used?

The downloaded OpenXava zip file has a configured workspace to use with Eclipse. OpenXava is not using Eclipse resources but it's based in the Ant Tool (which it's open source). You can use whatever IDE that you want without problem or also from command lines.

What JDK need to run?

The OpenXava works with JDK 1.4.2, JDK 5.0 and JDK 6.0.
If you want to use EJB3 JPA you need at least Java 5.
JDK 1.3.1 works fine with OpenXava 1.2.1, but it's not tested since OpenXava 2.

Could we generate a non J2EE application using OpenXava?

It would be easy adapt OpenXava to do it because the OpenXava kernel it's XML (also POJOs in v3) and it hasn't strong links to EJB, Hibernate or Servlets (not as EJB3 + JSF). To construct a generator to use with PDAs or two tiers applications (by example using swing) it's easy. The initial OpenXava version was working with Swing + EJB. The latest one is working with JSP + Hibernate. All of these without changes in your OpenXava applications. The OpenXava applications have always the same structure.
With something of inventiveness you can even develop a .NET, PHP or Ruby version.

What licence type has OpenXava?

LGPL. You can use OpenXava to develop every comercial project than you want. The unique forbidden thing is developing your own comercial version of OpenXava.
