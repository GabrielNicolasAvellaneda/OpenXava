¿Por qué es mejor OpenXava que IBM WebSphere Studio o Rational Application Developer?

Es cierto que estas herramientas de IBM son sofisticadas y pueden suponer una buena ayuda en muchas circunstancias, pero OpenXava se presenta como una alternativa, especialmente cuando se quiere desarrollar aplicaciones de Gestión.
Algunas de las ventajas de OpenXava sobre WebSphere Studio y Rational Application Developer son:
  • OpenXava permite desarrollar más rápidamente y de forma más sencilla. Prueba hacer el ejemplo que viene con el tutorial de OpenXava, con el mismo nivel de funcionalidad, y usando la misma tecnología (J2EE incluyendo EJB CMP) con WebSphere Studio y verás.
  • OpenXava es un generador activo de código mientras que WebSphere Studio es un generador pasivo de código.
  • El coste de licencia por desarrollador de OpenXava es de 0 euros, mientras que el de WebSphere Studio...
  • OpenXava genera aplicaciones para JBoss y WebSphere (fácilmente extensible a otros), WebSphere Studio genera aplicaciones solo para WebSphere Application Server.

¿Por qué es mejor OpenXava que una solución MDA?

Yo no diría que es mejor, simplemente es diferente. Con MDA generamos una aplicación a partir de unos modelos UML y así reducimos los costes de desarrollo. Con OpenXava hacemos lo mismo, pero en lugar de hacer la definición con UML la hacemos con XML.
Y así tenemos un conjunto de suculentas ventajas respecto a MDA:
  • XML es texto, por lo tanto:
    • No necesitamos herramientas sofisticadas, un simple editor sirve.
    • Mucha gente prefiere el texto a los gráficos. Porque es verdad 'que una imagen vale más que mil palabras', pero también es verdad que para la humanidad la palabra (texto) ha sido un medio muy eficaz de transmitir información durante siglos (al final el Latín se impuso a la escritura jeroglífica egipcia)
  • XML es por naturaleza extensible, así añadir nuevas caracteristicas a OpenXava es muy fácil.
  • XML esta muy extendido y cuenta con gran cantidad de validadores y editores especiales.
  • XML es un estándar conocido.
  • XML es muy fácil.
OpenXava y MDA no son cosas excluyentes, de hecho se puede crear una herramienta MDA encima de OpenXava; solo habría que generar los archivos XML de OpenXava a partir del modelo UML.

¿Que diferencias hay entre OpenXava y XDoclet?

XDoclet consigue que lo que antes se hacía con varias clases de Java y varios complicados archivos XML ahora se puede hacer con una sola clase de Java.
OpenXava consigue que lo que antes se hacía con varias clases de Java y varios complicados archivos XML ahora se puede hacer con un solo y simple archivo XML.
XDoclet y OpenXava son parecidos, solo que en XDoclet la base para definir nuestras aplicaciones es el código Java y en OpenXava es el XML.
XML parece más natural para definir que Java y sus anotaciones porque:
  • Permite estructuras jerarquicas.
  • Validación y autocompletado automático gracias a los editores XML.
  • Más fácil de aprender por un programador no Java.
Otra diferencia entre XDoclet y OpenXava es que OpenXava genera no solo el código EJB, sino una aplicación entera, con la capa del controlador y de la interfaz gráfica, y ya lista para usar.
XDoclet y OpenXava no son exclusivos, de hecho OpenXava utiliza XDoclet para generar el código EJB, y además podemos escribir beans de EJB con XDoclet e integrarlos en una aplicación OpenXava si queremos.

¿Por qué OpenXava no usa Struts?

Struts permite definir la parte de controlador de una aplicación web, y OpenXava ya tiene su propio marco de controlador. Struts está orientado a la navegación entre páginas, mientras que en las aplicaciones típicas de gestión suele haber muchos casos en los que permanecemos en una misma página en la que el usuario puede hacer una cantidad grande de acciones; para esto Struts es un poco engorroso, y la capa de controlador de OpenXava acopla mejor.
Otra ventaja es que la capa de controlador de OpenXava no depende de la tecnología web, y así podríamos tener, por ejemplo, una aplicación OpenXava con otra tecnología de presentación (de hecho existe una versión de OpenXava para hacer aplicaciones Swing).
Programadores con experiencia en Struts han mostrado alivio al trabajar con OpenXava para hacer aplicaciones de gestión.

¿Con qué sistema de gestión de usuarios y navegación cuenta OpenXava?

Las primeras versiones de OpenXava (en swing) tenían incluido un sistema de seguridad y navegación. La versión actual no lo hace, en su lugar genera los módulos web que el desarrollador puede incrustar en el marco de seguridad y navegación que desee.
Una opción buena es usar los modulos OpenXava como portlets e integrarlos con un portal (como JetSpeed o Websphere Portal). Así es como la mayoría de aplicaciones actuales OpenXava se instalan en los clientes.

¿Qué servidores de aplicaciones están soportados?

OpenXava soporta:
  • JBoss 3.2.x y 4.0.x
  • WebSphere 5.0, 5.1 y 6.0
Usando EJB CMP2 nativo para persistencia.
Adaptarlo para que funcione con otros servidores J2EE es harto sencillo.

¿Qué herramientas de desarrollo se pueden usar?

El archivo de descarga de OpenXava viene con un workspace para Eclipse 3 configurado. Pero OpenXava no usa recursos de Eclipse sino que está basado en la herramienta de código abierto 'Ant', por lo que sin ningún problema se puede usar con cualquier otro IDE, e incluso desde línea de órdenes.

¿Con qué navegadores de internet funciona?

Internet Explorer, Mozilla y Firefox. El nivel de JavaScript que usa es bajo, y la disposición no se hace con CSS, así que funciona bien con un abanico grande de navegadores.

¿Qué JDK necesita para funcionar?

OpenXava funciona con JDK1.3.1, JDK1.4.2 y JDK1.5.0

¿Podemos generar con OpenXava una aplicación que no sea J2EE?

No sería difícil de adaptar, porque el nucleo de OpenXava es el XML, y no está vinculado muy fuertemente a EJB o Servlets. Así que hacer un generador que hiciera aplicaciones para PDAs, o aplicaciones de 2 capas (usando swing por ejemplo) es sencillo. De hecho la versión inicial de OpenXava funcionaba con Swing + EJB, la actual funciona con JSP + EJB, y posiblemente la versión 2 funcione con JSP + Hibernate. Y todo esto sin cambiar las aplicaciones OpenXava que siempre tienen la misma estructura.
Con un poco de ingenio se podría hacer incluso una versión .NET.

¿Qué tipo de licencia tiene OpenXava?

LGPL. Se puede usar OpenXava para desarrollar proyectos comerciales de cualquier tipo. Lo único es que no se puede hacer una versión comercial del propio OpenXava.
