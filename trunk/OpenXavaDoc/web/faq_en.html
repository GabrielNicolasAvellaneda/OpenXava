Why OpenXava is better than IBM WebSphere Studio or Rational Application Developer?

It's true that these tools from IBM are sophisticated and  they can be a good help in many circumstances, but OpenXava is a good alternative, specially when you want developing business applications.
Some advantages of OpenXava versus WebSphere Studio and Rational Application Developer are:
  • OpenXava allows you developing more rapid and easily. Test yourself, try to develop the example that comes with the OpenXava tutorial using WebSphere studio. With the same level of functionality and using the same technology (J2EE including EJB CMP), and you will see it.
  • OpenXava is a active code generator whereas WebSphere Studio is a passive code generator.
  • The price: OpenXava cost is 0 euros per developer, WebSphere cost is...
  • OpenXava generates applications for JBoss and WebSphere (easily extending to others) whereas WebSphere Studio only generates for WebSphere Application Server.

Why OpenXava it's better than an MDA solution?

We can talk about it's different and not only better. With MDA we can generate an application from UML models and in that way we reduce the costs involved in our Development. In fact with OpenXava we make the same thing but we don't start from UML models we start from XML.

Now we have advantages from this point of view beside the MDA model.

  • XML is text , the advantages are:
    • We don't need sofisticated tools, a single editor works fine.
    • There are many people than prefers text to graphical. Yes, it's true, one image is better than thousand words but is also true that the text has been a very efficient way to send information during centuries (at last the Latin was imposed to the ancient egyptian art of writing).
  • XML is extensible by nature, it's very easy to add new features to OpenXava
  • XML is very popular and has a lot of validators and special editors.
  • XML is a very known standard.
  • XML is very easy.
OpenXava and MDA aren't exclusive things, in fact you can create a MDA tool over OpenXava, only need to generate the XML files in OpenXava starting from the UML model.

What differences are between OpenXava and XDoclet?

XDoclet is very useful because with only one Java class obtain the same things than before using several Java classes and complex XML files.
OpenXava obtains the same from the point of view of one single XML file. Before OpenXava you needed several Java classes and complex XML files.
XDoclet and OpenXava are similar, the main difference it's than in OpenXava you use XML and in XDoclet you use Java code.
XML seems more natural to define Java and his logs because:

  • Permit hierarchical structures.
  • Validity and automatic completion using XML editors..
  • It's easier to learn by a non Java Programmer.
The other big difference between XDoclet and OpenXava is that OpenXava generates a full application, not only generates EJB code, with control and graphical interface layers. And of course ready to use.
XDoclet and OpenXava aren't opposite products, in fact OpenXava is using XDoclet to generate EJB code. If you want you can write EJB beans with XDoclet and use it in OpenXava application.

Why OpenXava don't use struts?

Struts allow to define the control layer of a web application, OpenXava has his own control framework. Struts is navigation between pages oriented, while in the typical business applications the user can made a lot of actions in the same page. Strut for these last thing is something complex, and the OpenXava control layer has been constructed to do so.

Another advantage from OpenXava is that control layer isn't dependable from web technologies, we could have by example one OpenXava application with a different presentation technology (in fact there is a OpenXava version to develop Swing applications).

Many Struts Advanced Programmers are very exciting working with OpenXava to develop business applications.

What type of users manager system and navigation could you use with OpenXava?

The initial OpenXava versions (using swing) had a security system and navigation system. The latest version don't do it. These last version generate a web module and the developer can crust in the security or navigation framework to choose. A good option could be use OpenXava modules as portlets and using it integrated with a portal (like JetSpeed or Websphere Portal). This is the way to install in our customers the OpenXava applications.

What Application Servers are supported?

OpenXava supports:
  •  JBoss 3.2.x and 4.0.x
  • WebSphere 5.0, 5.1 and 6.0
Using native EJB CMP2 for persistence.
The work to adapt OpenXava to make it compliant with another J2EE server is easy.

What Development Tools can be used?

The downloaded OpenXava zip file has a configures workspace to use with Eclipse 3. OpenXava is not using Eclipse resources but it's based in the Ant Tool (which it's open source). You can use whatever IDE that you want without problem or also from command lines.

What Internet Browsers are supported?

Internet Explorer, Mozilla and Firefox. The JavaScript level used is low and we don't use CSS for layout. It's working fine with many browsers

What JDK need to run?

The OpenXava works with JDK1.3.1, JDK 1.4.2 and JDK 1.5.0

Could we generate a non J2EE application using OpenXava?

It would be easy adapt OpenXava to do it because the OpenXava kernel it's XML and hasn't strong links to EJB and Servlets. To construct a generator to use with PDAs or two tiers applications (by example using swing) it's easy. The initial OpenXava version was working with swing plus EJB. The latest one is working with JSP plus EJB and it's possibly that version 2 will work with JSP plus Hibernate. All of these without changes in your OpenXava applications. The OpenXava applications have always the same structure.
With something of inventiveness you can develop a .NET version.

What licence type has OpenXava?

LGPL. You can use OpenXava to develop every comercial project than you want. The unique forbidden thing is developing your own comercial version of OpenXava.
