Why to use OpenXava if we have Spring, Hibernate, JSF, EJB3 and so on?

Nowadays we live great times. It's true that Spring, Hibernate, JSF and EJB3 are far better than J2EE 1.4. All this frameworks and technologies are good (in fact OpenXava uses some of they), but if you need to develop business application the amount of code to write continues being big, and the productivity is low comparing with other technologies (Visual Basic, 4GL, Delphi, RPG, etc).
OpenXava simply has a higher level of abstraction in order to obtain a simple and easy way to develop business application with J2EE in a Object Oriented fashion.
If you do not believe me. Try it. See the demo applications, for example the JBoss Seam Booking Demo developed with OpenXava, see the code for it, and try to develop it with your favourite framework, and compare  yourself.

Why OpenXava is better than IBM WebSphere Studio or Rational Application Developer?

It's true that these tools from IBM are sophisticated and  they can be a good help in many circumstances, but OpenXava is a good alternative, specially when you want developing business applications.
Some advantages of OpenXava versus WebSphere Studio and Rational Application Developer are:
  • OpenXava allows you developing more rapid and easily. Test yourself, try to develop the example that comes with the OpenXava tutorial using WebSphere studio. With the same level of functionality and using the same technology (J2EE including EJB CMP), and you will see it.
  • OpenXava is a active code generator whereas WebSphere Studio is a passive code generator.
  • The price: OpenXava cost is 0 euros per developer, WebSphere cost is...
  • OpenXava generates applications for JBoss and WebSphere (easily extending to others) whereas WebSphere Studio only generates for WebSphere Application Server.

Why OpenXava is better than a MDA solution?

We can talk about it's different and not only better. With MDA we can generate an application from UML models and in that way we reduce the costs involved in our Development. In fact with OpenXava we make the same thing but we don't start from UML models we start from XML.

Now we have advantages from this point of view beside the MDA model.

  • XML is text , the advantages are:
    • We don't need sofisticated tools, a single editor works fine.
    • There are many people than prefers text to graphical. Yes, it's true, one image is better than thousand words but is also true that the text has been a very efficient way to send information during centuries (at last the Latin was imposed to the ancient egyptian art of writing).
  • XML is extensible by nature, it's very easy to add new features to OpenXava
  • XML is very popular and has a lot of validators and special editors.
  • XML is a very known standard.
  • XML is very easy.
OpenXava and MDA aren't exclusive things, in fact you can create a MDA tool over OpenXava, only need to generate the XML files in OpenXava starting from the UML model.

What differences are between OpenXava and XDoclet?

XDoclet is very useful because with only one Java class obtain the same things than before using several Java classes and complex XML files.
OpenXava obtains the same from the point of view of one single XML file. Before OpenXava you needed several Java classes and complex XML files.
XDoclet and OpenXava are similar, the main difference it's than in OpenXava you use XML and in XDoclet you use Java code.
XML seems more natural to define Java and his logs because:

  • Permit hierarchical structures.
  • Validity and automatic completion using XML editors..
  • It's easier to learn by a non Java Programmer.
The other big difference between XDoclet and OpenXava is that OpenXava generates a full application, not only generates EJB code, with control and graphical interface layers. And of course ready to use.
XDoclet and OpenXava aren't opposite products, in fact OpenXava is using XDoclet to generate EJB code. If you want you can write EJB beans with XDoclet and use it in OpenXava application.

Why OpenXava does not use struts?

Struts allow to define the control layer of a web application, OpenXava has his own control framework. Struts is navigation between pages oriented, while in the typical business applications the user can made a lot of actions in the same page. Struts for these last thing is something complex, and the OpenXava control layer has been constructed to do so.

Another advantage from OpenXava is that control layer isn't dependable from web technologies, we could have by example one OpenXava application with a different presentation technology (in fact there is an OpenXava version to develop Swing applications).

Many Struts Advanced Programmers are very exciting working with OpenXava to develop business applications.

It's only XML, no Java class?

OpenXava uses JavaBeans to insert your custom functionality in the application using the strategy pattern.

How scalable it is?

OpenXava generates a standard J2EE application, hence it's as scalable as J2EE.
OpenXava uses Hibernate or EJB CMP2 for persistence, and some session scope state at servlets level. The code generated by OpenXava is good, but when you write your own code you must to be wise in order to obtain a good performance, specially in batch processes, but this is an issue of J2EE not OpenXava.

What type of users manager system and navigation could you use with OpenXava?

The modules generated by OpenXava are JSR-168 portlets, hence are deployabled in any compliant Java Portal with its own security and navigation features. 
OpenXava is bundle with a JetSpeed 2 and uses the security and navigation of JetSpeed 2, but you can choose another portal or create your own security and navigation framework for OpenXava modules.

What Application Servers are supported?

OpenXava has been tested with:
  • Tomcat 5.5
  • JBoss 4.0.x and 3.2.x
  • WebSphere 6.0, 5.1 and 5.0
Any Servlet 2.3 compliant container can execute OpenXava applications.
If you want use EJB (optional in v2) you need at least a J2EE 1.3 application server.

What Java Portals are supported?

Any JSR-168 compatible portal. Although we use in production the next ones:
  • Apache JetSpeed 2
  • WebSphere Portal 5.1
  • Liferay

What Development Tools can be used?

The downloaded OpenXava zip file has a configured workspace to use with Eclipse 3.1. OpenXava is not using Eclipse resources but it's based in the Ant Tool (which it's open source). You can use whatever IDE that you want without problem or also from command lines.

What Internet Browsers are supported?

Internet Explorer, Mozilla and Firefox.

What JDK need to run?

The OpenXava works with JDK 1.3.1, JDK 1.4.2 and JDK 5.0

Could we generate a non J2EE application using OpenXava?

It would be easy adapt OpenXava to do it because the OpenXava kernel it's XML and hasn't strong links to EJB, Hibernate or Servlets (not as EJB3 + JSF). To construct a generator to use with PDAs or two tiers applications (by example using swing) it's easy. The initial OpenXava version was working with swing plus EJB. The latest one is working with JSP plus Hibernate. All of these without changes in your OpenXava applications. The OpenXava applications have always the same structure.
With something of inventiveness you can even develop a .NET version.

What licence type has OpenXava?

LGPL. You can use OpenXava to develop every comercial project than you want. The unique forbidden thing is developing your own comercial version of OpenXava.
